# Сценарии

NOUZ полезен в разных ситуациях: когда база знаний только создаётся, и когда она уже большая и местами неоднородная. Это разные режимы работы.

## Кому подходит

NOUZ имеет смысл, если база знаний уже стала больше обычной папки с заметками: в ней есть домены, уровни, повторяющиеся темы, решения, логи, проектная документация или рабочие материалы.

Подходит:

- разработчикам и командам, которым нужен MCP-доступ к проектной памяти, которая постоянно меняется;
- владельцам Obsidian-базы (`vault`) или другой папки Markdown, где часть связей уже есть, но многое осталось только “в голове”;
- базам, где важно различать структуру и содержание: что заметка должна была означать и о чём она фактически говорит.

Не лучший вариант:

- для маленькой базы из 20–30 заметок без устойчивых тем;
- если нужен только полнотекстовый поиск;
- если нет готовности один раз настроить домены и проверить качество эталонов.

## Новая база

**Ситуация:** вы начинаете базу с нуля или переносите небольшой набор материалов. Структура ещё не заросла случайными связями, поэтому NOUZ можно подключать мягко: граф растёт вместе с базой.

**Как работать:**

1. Начать с LUCA: YAML, родители, навигация по графу.
2. Добавлять новые заметки обычным способом.
3. Для новых заметок вызывать `suggest_metadata` (предложить разметку) или `suggest_parents` (предложить родителей).
4. Когда накопится достаточно материала, включить PRIZMA: эталоны, эмбеддинги, мосты и `core_mix` — доменный состав узлов.

Минимальная заметка:

```yaml
---
type: заметка
level: 4
parents:
  - "[[Название модуля]]"
parents_meta:
  - entity: Название модуля
    link_type: hierarchy
tags:
  - тема
---
```

Что получает агент:

```text
suggest_metadata("Новая заметка.md")
→ {
    sign: "E",
    level: 4,
    suggested_parents: ["Проект/Архитектура.md"],
    bridges: [
      {type: "semantic", target: "Проект/Решения/ADR-001.md", cosine: 0.74, proposed: true}
    ]
  }
```

**Идея:** вы задаёте структуру по мере роста базы, а NOUZ предлагает место, домен, теги и мосты. Решение остаётся за вами.

## Существующая база

**Ситуация:** у вас уже есть большая база: сотни заметок, старые папки, разные стили записи, частично заполненный YAML. Здесь нельзя просто включить автоматику и доверить ей всё сразу. Настройки сильно зависят от реальных данных.

**Как работать:**

1. Сначала проиндексировать базу без массовых правок.
2. Посмотреть реальные домены, типы заметок, уровни и разрыв между папками и содержанием.
3. Написать эталоны под эту базу, а не под абстрактный пример.
4. Запустить `calibrate_cores` и проверить сырые / mean-centered косинусы.
5. Настроить пороги классификации и связей: `sign_spread`, `pattern_second_sign_threshold`, `semantic_bridge_threshold`, `parent_link_threshold`.
6. Запускать `recalc_signs`, `suggest_metadata` и `process_orphans` партиями, проверяя результаты.

Пример:

```bash
index_all()
calibrate_cores()
recalc_signs()
suggest_metadata("Новый регламент мониторинга.md")
```

```text
bridges:
  - semantic: "Проект/Мониторинг.md" (cosine: 0.71)
  - semantic: "Проект/Инциденты/2026-04.md" (cosine: 0.63)
```

**Идея:** NOUZ показывает состояние уже существующей базы: потерянные связи, неразмеченные заметки, слабые родители, дрифт между `sign` и `core_mix`. Новая структура появляется после вашего решения, а не вместо него.

## Дрифт: когда структура расходится с реальностью

**Ситуация:** модуль «Мониторинг» начинался как набор технических инструкций, но со временем в нём появились инциденты, регламенты дежурств и решения по процессам. Название и YAML говорят одно, содержимое уже шире.

```python
recalc_core_mix(dry_run=True)
# → sign: E (Engineering), core_mix: {S: 61%, E: 39%}
# → предупреждение о дрифте: sign=E, core_mix показывает доминанту S
```

`core_drift` показывает расхождение между замыслом и фактическим содержанием. Это повод посмотреть на модуль внимательнее: переименовать, разделить на два, перенести часть заметок или принять новую траекторию.

## AI-агент: структурная память

**Ситуация:** агенту нужен графовый контекст: где заметка находится, с чем связана и какие материалы лежат вокруг неё.

```json
{
  "mcpServers": {
    "nouz": {
      "command": "python",
      "args": ["-m", "nouz_mcp"],
      "env": {
        "OBSIDIAN_ROOT": "/path/to/vault",
        "EMBED_API_URL": "http://127.0.0.1:1234/v1"
      }
    }
  }
}
```

```text
Пользователь: "Расскажи про архитектуру мониторинга"

Агент использует NOUZ:
1. list_files(subfolder="infrastructure") → 12 файлов
2. get_children("Infrastructure/Мониторинг.md") → [Prometheus.md, Grafana.md, Alerting.md]
3. get_parents("Infrastructure/Prometheus.md") → [Мониторинг.md]
4. read_file каждого потомка → полный контекст
```

Агент отвечает по структуре базы: использует иерархию, соседние узлы, дочерние материалы и близкие совпадения в поиске по эмбеддингам.

## Документация действующего проекта

**Ситуация:** в проекте есть архитектура, решения, баги, ретроспективы, инструкции и логи. Всё это часто лежит рядом, но не всегда связано.

```text
Проект/
├── Архитектура.md       # L3 модуль
├── Решения/
│   ├── ADR-001.md       # L4 заметка
│   ├── ADR-002.md
├── Баги/
│   ├── Bug-tracker.md   # L3 модуль
│   ├── Issue-42.md      # L4 заметка
└── Ретроспективы/
    └── 2026-Q1.md       # L5 артефакт
```

`get_children("Проект/Архитектура.md")` даёт агенту иерархию. `suggest_parents("Новый баг.md")` ищет похожие прошлые случаи. `list_files(level=4)` показывает ключевые заметки, из которых складывается проект.

## Что остаётся у вас

NOUZ считает, сравнивает и предлагает, но структура базы остаётся вашим решением. Это особенно важно для архивов, которые продолжают пополняться: сервер делает видимыми уровни, связи, домены и дрифт, а вы выбираете, что принять, что разделить и какие мосты оставить в графе.

Дальше обычно идут три шага: настроить эталоны под свою базу, проверить их качество через `calibrate_cores`, затем подключить MCP-клиент и дать агенту доступ к графовому контексту.

## Если база не в Obsidian

Obsidian — самый простой способ попробовать NOUZ. Тот же подход переносится и в другие рабочие среды: Notion, Confluence, Google Docs, GitHub или внутренний портал. Через MCP-коннекторы или отдельные адаптеры NOUZ работает рядом с текущей базой как структурный слой, а привычный инструмент остаётся на месте.

В рабочих базах задача обычно шире разовой установки. Сначала выбирается небольшой участок и проверяется, где накопились дубли, устаревшие страницы, слабые связи, изолированные разделы и места с недостающим контекстом для агента. После этого настраиваются домены, правила разметки, локальный индекс и порядок проверки предложений.
