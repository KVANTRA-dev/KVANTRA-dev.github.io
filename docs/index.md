## Что делает NOUZ

<div class="lead-panel">
  <p>NOUZ превращает Obsidian-базу в локальную карту знаний. Агент видит не только файлы, но и связи, уровни, соседний контекст и место заметки в графе. Структура остаётся под вашим контролем, а NOUZ помогает поддерживать порядок в базе и готовить точный контекст для ИИ-агента.</p>
</div>

## Зачем это нужно

Когда база растёт, агенту всё труднее выбрать нужный контекст: он читает лишние файлы, пропускает важные связи, путает черновики с выводами и каждый раз собирает материал заново. NOUZ даёт ему аккуратные действия над базой: прочитать заметку, увидеть её место в графе, пройти по связям, предложить разметку и найти похожие материалы.

## Что меняется для агента

| До NOUZ | С NOUZ |
| ------- | ------ |
| Агент читает папку почти вслепую | Агент запрашивает конкретную заметку, родителей, детей и соседний контекст |
| Связи между материалами держатся в голове | Связи становятся частью локального графа и доступны через MCP |
| Старые страницы, дубли и сироты всплывают случайно | Сервер показывает слабые места и предлагает, что проверить |
| Любая автоматизация выглядит рискованной | Сначала предложения и предпросмотр, затем ваше решение |

## Не только Obsidian

Проще всего попробовать NOUZ на Obsidian, но сам подход шире Markdown-папки. Если материалы уже лежат в Notion, Confluence, Google Docs или GitHub, NOUZ можно подключать рядом с ними через MCP-коннекторы или отдельные адаптеры.

Такой переход обычно начинается с маленького участка базы: проверить структуру, увидеть, где агенту не хватает контекста, и только потом решать, какие правила и связи стоит закреплять.

## Данные и контроль

NOUZ работает локально: заметки остаются в вашей папке, индекс хранится в SQLite рядом с базой, а изменения в файлах проходят через явные инструменты. Можно начать с чтения и индексации, затем смотреть предложения в режиме предпросмотра и только после проверки записывать YAML-разметку.

Если используется локальный провайдер эмбеддингов, тексты не покидают машину. При облачном провайдере наружу отправляются только фрагменты, для которых запрошен эмбеддинг.

## Техническая основа

Граф строится сверху вниз. Каждый уровень отвечает за свою роль, а для больших баз можно добавить `meta_root` — якорный узел уровня 0, который собирает домены в одну систему и не участвует в семантических расчётах.

| Уровень | Тип | Что сюда помещать | Откуда знак |
| ------- | --- | ----------------- | ----------- |
| L1 | Ядро | Домен знаний — широкая область | Вручную или по эталонам |
| L2 | Паттерн | Тематическая рамка для дочерних модулей и заметок | Вручную или по эталонам |
| L3 | Модуль | Функциональная группа, проект | Вручную или по эталонам |
| L4 | Квант | Одна законченная мысль: гайд, решение, заметка, описание процесса | Собственный тип артефакта + домен по содержанию |
| L5 | Артефакт | Сырьё: логи, чаты, конфиги | Тип по эвристике содержимого |

## Быстро попробовать

```bash
pip install nouz-mcp
```

Проще всего начать с Obsidian. Укажите путь к папке базы (`vault`) и подключите NOUZ к MCP-клиенту. Режим **LUCA** работает без эмбеддингов; для PRIZMA и SLOI добавьте `config.yaml` и адрес сервиса эмбеддингов.

Пример режима с семантическим слоем:

```yaml
mode: prizma
```

Переменные окружения:

```bash
export OBSIDIAN_ROOT=/path/to/vault
export EMBED_API_URL=http://127.0.0.1:1234/v1
```

Запустите через MCP-клиент:

```json
{
  "mcpServers": {
    "nouz": {
      "command": "python",
      "args": ["-m", "nouz_mcp"],
      "env": {
        "OBSIDIAN_ROOT": "/path/to/vault"
      }
    }
  }
}
```
